Q. 3. ï¿½ï¿½Consider a pipelined processor than can issue up to one instruction per cycle.  For a certain program, 12% of its instructions are control transfer instructions. There is a history-based branch predictor, based on a branch target buffer, exactly as described in the lectures. The branch prediction accuracy of this program is 94%, and the branch penalty is 2 cycles. Assume that there no other pipeline bubbles, other than those from mispredicted branches.

(3 + 3 = 6 points)

(a)   What percentage faster would this program run if the branch penalty was 1 cycle instead of 2 cycles?

Assume the program has x instructions.

Then before the change, the run-time of the program is x + 0.12*0.06*x*2 cycles = 1.0144x cycles.ï¿½ //In this expression, the first term is the contribution of all the instructions without counting bubbles, vs the second term which adds the contribution of the bubbles.

After the change, the run-time of the program is x + 0.12*0.06*x*1 cycles = 1.0072x cycles.

This means the program is (1.0144-1.0072)/1.0144*100 % faster = 0.71% faster.

(b)  If the pipeline was 25 stages deep. with a branch penalty of 10 cycles, with everything else remaining the same, what percentage faster would program run if the branch penalty were reduced to 5 cycles?

Assume the program has x instructions.

Then before the change, the run-time of the program is x + 0.12*0.06*x*10 cycles = 1.072x cycles.ï¿½ //In this expression, the first term is the contribution of all the instructions without counting bubbles, vs the second term which adds the contribution of the bubbles.

After the change, the run-time of the program is x + 0.12*0.06*x*5 cycles = 1.036x cycles.

This means the program is (1.072-1.036)/1.072*100 % faster = 3.36% faster.

 

Q. 4 A 5-ï¿½stage single pipeline computer uses the pipeline mentioned in the lecture. Unlike what was discussed in class, it resolves conditional branch instruction at the end of its third pipeline stage (EX), instead of the second stage (DE). A branch is said to be resolved at a certain stage if its direction and target are both computed by that stage.

(3 + 3 = 6 points)

(a)   What is the most likely action(s) that would need to be moved to the 3rd stage from the 2nd stage, that would cause this change from the lectures?

Branch resolution in DLX is simple for unconditional control transfer instructions (CTIs), since only the target needs to be computed, which can be computed with an adder in the DE stage, even before the instruction is decoded, likely without slowing down the DE stage.

However, branch resolution for conditional CTIs needs two actions: fetching the source register, and then performing the comparison check (such as equal to zero). Both these operations may not fit in the DE stage without making it longer. So, the designer may choose to move the comparison check to the EX stage, to avoid delaying the DE stage.

(b)  Why might a hardware designer choose to design a pipeline where branches are resolved in the third stage instead of second? In other words, what harm might come from resolving branches earlier in the second stage?

As per the reasoning in part (a) above, the designer may not want to resolve branches earlier in the DE stage, because doing so may need to make the DE cycle take longer, thus slowing down the clock for all instructions. This may hurt the pipeline far more than the extra branch penalty for moving the comparison to the EX stage.
